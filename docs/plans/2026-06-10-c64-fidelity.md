## Overview

Two C64-fidelity gaps, both of the same shape: a correct lower layer (VC/RC bad-line machinery; IEC command/channel FSM) topped by an output stage that ignores it (a fixed-origin renderer; a structurally dead bit-level handshake), with documentation that claims more than the code delivers. The plan fixes each output stage by **driving it from the already-correct lower layer**, adds unit evidence that needs no copyrighted ROMs, pins system-level behaviour with data-gated golden tests, and restores honest docs as its own first deliverable in each track.

All file references below are against the current tree at commit `42b0729`.

- Meanwhile `bad_line_condition()` (lines 704-707) *does* honour `modes_.yscroll`, and `advance_video_counters` (lines 193-219) maintains a faithful Bauer §3.7.2 VC/VCBASE/RC/VMLI/display-state machine per cycle. So the CPU-visible timing (BA stalls, bad-line cadence) and the pixels disagree whenever YSCROLL ≠ 3 — exactly what every smooth-scrolling game does.
- The header (`vic_ii_6569.hpp:37-42`) claims only "strictly cycle-exact beam timing" is deferred — false while XSCROLL/YSCROLL/CSEL/RSEL never reach a pixel.
- `NOTES.md:41-48` ("Still deferred") is stale in the *opposite* direction: it still lists bitmap modes, ECM, the sprite engine, collisions, and save/load state as deferred, all of which are implemented (`render_line` covers all five modes at lines 492-559; sprites + collisions at 568-698; `save_state/load_state` at 719-761).

### 1.2 Architecture decision: drive the renderer from VC/RC (recommended), not from offsets

**Recommendation: latch the VC/RC machinery's per-line outputs and render from them.**

Mechanism: `advance_video_counters` already runs at cycle index 13 of every line, reloading `vc_` from `vcbase_` and arming `display_state_`/`rc_` on a bad line (lines 196-203). Add three per-line render latches captured at that same point:

```
line_vc_      // vc_ right after the cycle-13 reload  → first video-matrix cell of this line
line_rc_      // rc_ right after the cycle-13 reload  → glyph row 0..7 for this line
line_display_ // display_state_                        → display vs idle line
```

`render_line(y)` is invoked from `tick` when `raster_x_` wraps (line 330), i.e. *after* the completed line's cycle-57 RC increment/VCBASE latch has run — so it cannot use the live `vc_`/`rc_`. The latches are still valid at that point (the next line's cycle 13 has not yet executed), which makes this a clean, ordering-safe handoff. The renderer then computes:

- cell index = `(line_vc_ + text_col) & 0x3FF` (replacing `text_row * text_columns + text_col` at line 480-481),
- `glyph_y = line_rc_` (replacing `char_row % 8`),
- idle lines (`!line_display_` inside the open border window) render the idle pattern (see 1.4).

**Why not offset-based** (`text_row = (y - (0x30 + yscroll + ...)) / 8` style): it duplicates the bad-line→row logic the chip already has, must special-case the DEN latch, FLD, and mid-frame YSCROLL writes, and re-creates the exact "two sources of truth disagree" defect under repair. The VC/RC path costs three scalar copies per line, makes YSCROLL pixel behaviour *fall out* of the already-tested bad-line logic, and gives line-granularity FLD/linecrunch effects (common in the target games) approximately-correct rendering for free.

**YSCROLL semantics under this model** (matches hardware at line granularity): the vertical border window is *fixed* by RSEL; YSCROLL only moves the bad lines, hence where RC=0 rows begin. With the boot default `$D011 = $1B` (YSCROLL=3, RSEL=1, DEN=1), the first bad line is raster `$33` = 51 = today's `display_top`, so the default screen is **pixel-identical** to the current renderer — the golden boot hash must not change. This is the central no-regression gate.

### 1.3 Border window (CSEL/RSEL) and XSCROLL

Border masking is layered *on top of* graphics layout (38-column mode crops, it does not re-layout — true to hardware):

| | open | close (exclusive) |
|---|---|---|
| RSEL=1 | line 51 (`$33`) | 251 (`$FB`) |
| RSEL=0 | line 55 (`$37`) | 247 (`$F7`) |
| CSEL=1 | X 24 | 344 |
| CSEL=0 | X 31 | 335 |

(Bauer §3.9; the current model's `display_left = 24` already coincides with the hardware X coordinate for CSEL=1, so CSEL=0 is a pure mask `[24,31) ∪ [335,344)` over the unchanged 40-column layout.)

Implementation: a pure function of `y`, `modes_.rsel/csel`, and the existing `den_latched_line_30_` gate, evaluated in `render_line` — **not** a stored flip-flop. Rationale: no save-state format impact, deterministic, and the flip-flop's only observable advantage (open-border tricks) is an explicit non-goal. Document the limitation.

XSCROLL: shift graphics right by `modes_.xscroll` pixels — `disp_x = x - (24 + xscroll)`; window pixels with `disp_x < 0` (the left gap, up to 7 px) render `bg0` with `fg_mask_ = 0`, and columns past the 40th simply never appear because the border closes at the fixed right edge. With CSEL=0 the borders hide most of the gap — which is precisely why games pair `CSEL=0` with XSCROLL. Default `$D016 = $C8` → xscroll=0 → pixel-identical output.

Sprites: `render_sprites` plots after border fill and is unchanged; sprites continue to draw over border pixels (current behaviour, line 442 and 565). Hardware layers border *over* sprites; retaining the current layering is a documented non-goal (see §8).

### 1.4 Idle-state rendering

When a line inside the open window has `line_display_ == false` (e.g. FLD: YSCROLL written so no bad line fires), hardware displays the idle `$3FFF` fetch in black over the background. Proposed level of detail: render as hi-res data from `fetch(0x3FFF)` (`0x39FF` under ECM) with foreground colour black, background `bg0`, `fg_mask_` set on set bits. Cheap, uses the existing `fetch()` floating-bus latch correctly, and makes FLD look right at line granularity. (Open question #3 offers the simpler bg0-only alternative.)

## Work Item 2 — C1541 bit-level IEC handshake

### 2.1 Current state (grounded)

`src/chips/storage/c1541/synthetic_drive.cpp:169-213` (`serial_tick`) is non-functional on four axes:

1. **DATA is never released.** The drive pulls DATA on the ATN edge (line 180) and on byte completion (line 208) and there is no `pull(data, false)` anywhere. The KERNAL talker waits for the listener to *release* DATA ("ready-for-data") before every byte — so the protocol deadlocks at the first command byte.
2. **Self-sensing.** `sense()` (lines 161-163) reads `bus_->asserted(l)`, the wired-AND *including the drive's own driver*. Combined with (1), `!sense(data)` at line 196 is always false → every assembled bit is 0 even if the controller toggles DATA perfectly.
3. **`link_mode::talking` is set (line 183) but has no code path**; `serial_phase_` is declared (`synthetic_drive.hpp:97`), serialized (cpp:237/259), and never used. There is no turnaround, no byte-send, no EOI in either direction.
4. **First-edge swallow.** `atn_prev_` initializes/resets to `true` (hpp:100, cpp:48 — the comment "(released)" is itself wrong: the variable stores the *asserted* level). If the controller asserts ATN before the drive's first tick, the edge test at line 175 never fires and the drive never acks — the KERNAL's ~1 ms device-present probe then reports DEVICE NOT PRESENT.
5. Secondary defect to fix in passing: bits are sampled on the CLK *assert* edge (`clk && !clk_prev_`, line 195); the IEC data-valid point is the CLK **release** edge.

The command/channel layer above (`process_command`/`load_into_channel`, lines 59-141) is sound and covered by `tests/synthetic_drive_test.cpp` — but only via the `debug_*` bypass. The in-file claim at lines 166-168 ("end-to-end LOAD is validated with the C64 KERNAL ROM") and `NOTES.md:19-24` are stale: no such test exists (`tests/golden/` has only `c64_basic_boot_test.cpp`, which never touches IEC).

Wiring facts that shape the design: the drive is scheduled at divider 1 alongside the CPU (`src/manifests/c64/c64_runtime.hpp:64-72`, golden boot `tests/golden/c64_basic_boot_test.cpp:133-135`), so `serial_tick` runs once per φ2 cycle ≈ **1.0 µs/tick (PAL)** — microsecond timing can be counted in ticks. The C64 side drives ATN/CLK/DATA from CIA2 PA3-5 and senses CLK/DATA on PA6/7 (`c64_runtime.cpp:128-147`), with the KERNAL's software delay loops running on the emulated 6510 — both ends share the same clock, so protocol timing is internally consistent.

### 2.2 Architecture decision

**(a) Bus sampling: add a self-excluding read to `iec_bus`** (`src/chips/shared/iec_bus.hpp`):

```cpp
[[nodiscard]] bool asserted_by_others(std::uint8_t device, line l) const noexcept {
    return (pulled_[static_cast<std::size_t>(l)]
            & static_cast<std::uint16_t>(~(1U << (device & 0x0FU)))) != 0U;
}
```

Chosen over drive-side bookkeeping because the alternative ("release own pull, sample, re-pull") perturbs the bus mid-protocol, and shadow-tracking one's own drivers re-implements state the bus already holds in `pulled_`. It is a 4-line header addition with direct tests in `src/chips/shared/tests/iec_bus_test.cpp`. **`full_drive` must NOT switch to it** — its `via1_iec_input` (`full_drive.cpp:118-132`) models the physical input buffer, which on real hardware does see the drive's own pulls; the DOS ROM accounts for that.

**(b) State machine: per-tick sampled (polling) FSM, not edge-callback-driven.** The drive is already ticked every φ2 cycle in lockstep; a sampled FSM with tick-counted delays is deterministic, save-state friendly, and matches how the scheduler works (`src/runtime/scheduler.hpp` fixed-divider lockstep). Repurpose the existing `serial_phase_` + `link_` fields, add a `std::uint32_t serial_timer_` (ticks in current phase) and `bool eoi_pending_`.

**Listener path** (commands under ATN and filename payload use the same engine):

| phase | drive holds | waits for | timeout action |
|---|---|---|---|
| `l_hold` | DATA pulled | controller releases CLK (talker-ready) | — |
| `l_ready` | DATA released | CLK asserted within ~200 µs → `l_bits` | >200 µs → EOI: pull DATA ~60 µs, release, set `eoi_pending_`, wait CLK assert → `l_bits` |
| `l_bits` | nothing | sample DATA on each CLK **release** edge, LSB-first, bit = DATA released | — |
| after bit 8 | pull DATA (frame ack ≤1 ms budget; we ack in 1 tick) | dispatch to `process_command` (under ATN) / `process_listen_payload` | back to `l_hold` |

ATN handling becomes **level-triggered**: `if (atn && !under_atn_)` → abort whatever is in flight, enter command mode, pull DATA. This both fixes the `atn_prev_` initial-value swallow and implements the hardware rule that ATN preempts everything. ATN release keeps the current routing (talk-addressed → turnaround; listen-addressed → payload; else idle + `release_all`).

**Talker path** (after TALK+SA, on ATN release): turnaround — wait for controller to release CLK and pull DATA, then drive pulls CLK and releases DATA. Per byte:

| phase | action |
|---|---|
| `t_ready` | release CLK; wait for DATA released by controller (sensed via `asserted_by_others`) |
| `t_eoi` (last byte only) | do **not** pull CLK for >200 µs; wait for the listener's DATA ack pulse (pull→release) |
| `t_bits` | per bit: pull CLK + set DATA driver (bit 1 = released), hold ~35 µs; release CLK (data valid), hold ~35 µs — ~70 µs/bit, comfortably inside the KERNAL ACPTR tolerances (Ts/Tv ≥ 20 µs) |
| `t_ack` | pull CLK, release DATA; wait ≤1 ms for listener DATA pull; then next byte or (after the EOI byte) release CLK → idle |

File-not-found / empty channel: at turnaround the drive performs the turnaround CLK pull, then at `t_ready` **never asserts CLK again** for the first byte — the KERNAL's EOI/timeout path raises ST and reports FILE NOT FOUND. UNTALK/UNLISTEN under ATN → `link_ = idle`, `bus_->release_all(device_)`; this keeps `transfer_active()` (`synthetic_drive.hpp:55`) honest for the player adapter's `await_drive` autostart phase (`src/apps/player/adapters/c64/c64_adapter.cpp:349-364`).

All timing constants expressed as named tick counts (PAL φ2 ≈ 1 µs/tick; NTSC's 1.0227 MHz keeps every margin safe since the named IEC limits have ≥2× headroom).

## 7. Explicit non-goals

- Cycle-exact beam timing: per-cycle X splits, mid-line `$D020/$D021` raster splits at pixel exactness, sprite-DMA BA/AEC timing.
- Open-border / sprite-in-border tricks and true border flip-flop semantics (the pure-function window forecloses these by design, documented).
- Fastloaders, copy protection, JiffyDOS — that is `full_drive`'s mandate (`c1541/NOTES.md:26-40`); `full_drive` is untouched here.
- SAVE (drive-side write path), multiple drives / devices 9-11, GCR-level work in the synthetic drive.
- New scroller-demo golden corpora; system-level scrolling evidence is unit-test + unchanged-boot-hash only.
- Any change to the C64↔CIA2 IEC wiring or to the scheduler.
