## Goal

A functional Irem M72 board family driven from the player via an `irem_m72`
family id, with R-Type as the first game: NEC V30 main CPU, Z80 sound CPU
(already in tree), the M72 tilemap/sprite video unit, YM2151 + DAC audio, and
declarative per-game TOML manifests layered on a shared board manifest. No
existing console path changes except the additive extensions named in ADR 0012.

## Hard prerequisites (blockers, surfaced up front)

1. **Game ROM sets are never committed.** Board bring-up, boot, and parity
   tests skip (`SKIP_RETURN_CODE 4`) unless an env var points at a locally
   supplied R-Type ROM set. Functional CPU/loader/video work proceeds without
   them against synthetic fixtures.
2. **V30 conformance corpus** (recommended): the public per-instruction JSON
   corpora for the 8088/V20 family (SingleStepTests format, the same shape the
   Z80 singlestep harness consumes). Env-gated via `MNEMOS_V30_TESTS_DIR`,
   never committed.
3. **Reference emulators (MAME et al.) are licence-incompatible for code
   lifting** (GPL vs the Apache-2.0/MIT split, ADR 0003). They serve as
   behavioural references only; implementation is from public ISA / hardware
   documentation, gated by the conformance corpus (ADR 0006 policy).

## Phase A — V30 CPU core (`src/chips/cpu/v30`)

The reusable, system-agnostic piece. Buildable and testable with no board,
no ROMs — only public NEC V30 / 8086-family ISA documentation.

- **A1 (this increment):** chip skeleton + executable core — `icpu` subclass
  with the V30 programming model (AW/BW/CW/DW, SP/BP/IX/IY, the four segment
  registers, PC, PSW — surfaced under the conventional Intel names
  AX..DX/SI/DI/CS..SS/IP/FLAGS that every corpus and reference doc uses),
  little-endian word access composed over `ibus` byte reads, 20-bit physical
  addressing, `reset` (CS=FFFF, IP=0), prefix handling (segment override,
  REP/REPNE + the V-series REPC/REPNC, LOCK), modrm/EA decode, and the 8086
  base instruction map plus the 186/V-series additions M72 code uses (PUSHA/
  POPA, PUSH/IMUL imm, INS/OUTS, shift-by-imm, ENTER/LEAVE, BOUND). INTR/NMI
  dispatch (vector via injected ack callback), separate 16-bit I/O space via
  port callbacks (the Z80 pattern), HLT, save/load state, `introspection_surface`
  (register_view + trace_target), factory registration (`nec.v30`), unit tests
  per instruction group. First-order documented cycle costs; exact V30 timing
  is a later increment.
- **A2:** the V30 `0F`-prefix extension group (bit-field ops, BCD string ops,
  TEST1/SET1/CLR1/NOT1), plus the 8080-emulation-mode entry points stubbed as
  documented no-ops with operand-byte consumption (no M72 title uses them).
- **A3:** timing refinement — per-EA cycle table, V30-specific (not 8086)
  instruction timings, prefetch-queue first-order model if parity work needs it.
- **A4:** conformance harness polish: per-instruction undefined-flag masks,
  cycle-count comparison once A3 lands. The functional regs+memory compare
  harness itself ships in A1 (`MNEMOS_V30_TESTS_DIR`, skips when unset).

## Phase B — ROM-set loader + board manifest (`src/manifests/irem_m72`)

- Multi-file ROM regions in `manifests/common`: per-file offset/stride
  (even/odd interleave for the V30 program pair), per-file CRC32/SHA, zip
  bundle support composing the existing `compression` + `security/cryptography`
  modules. Unit-tested with synthetic fixtures.
- `m72.toml` board manifest: V30 + Z80 on their buses, work RAM, video RAM,
  sprite RAM, palette RAM, sound latch MMIO, I/O ports (inputs/DIPs/coin),
  program ROM banking callback. `games/rtype.toml` as the first game leaf.
- Little-endian wide bus accessors in topology (mirror of the BE ones from the
  SH-2/68000 hot-path work) — small standalone commit ahead of board wiring.
- First visual artifact: tile/sprite ROM regions exposed as introspection
  `debug_layer`s through the existing `--screenshot` path (raw decode viewer).

## Phase C — Video + IRQs (`src/chips/video/irem_m72_video`)

- Tilemap layers (two scrolling planes), sprites, per-tile priority,
  flip-screen, palette; 384×256 active raster around 55 Hz. Visible scanlines
  compose at beam-line start, so raster-time register changes affect later
  scanlines without repainting earlier lines.
- Scanline-compare raster IRQ + vblank IRQ into the V30 via the board's IRQ
  callback wiring; sprite DMA trigger.
- R-Type attract mode renders; screenshot-parity baseline established
  (data-gated on the ROM set).

## Phase D — Audio + inputs

- `src/chips/audio/ym2151` (reusable well beyond Irem) + `src/chips/audio/dac`;
  sound latch + Z80 INT/NMI plumbing; sample playback path with DAC writes
  mixed at the YM2151 sample boundary.
- Scheduler rational-divider extension for the YM2151's 3.579545 MHz crystal
  (32 MHz master: V30 = /4, Z80 = /8 stay exact integer dividers).
- Cabinet inputs (coin/start/service/test) + DIP banks: manifest `[input]` /
  `[dip]` schema, peripheral_sdk device, player key mapping. R-Type playable.
  The current ROM-set schema parses `[[dip]]` option tables with optional raw
  condition masks and the Irem adapter retains them; the checked-in true-M72
  roster carries per-family DIP tables, including conditional coinage and clone
  parent inheritance.

## Phase E — Roster + player polish

- Mr. Heli / Image Fight / Air Duel as data-only game manifests. Gated on the
  i8751 MCU question per ADR 0012 (`chips/cpu/mcs51`, or an explicitly marked
  per-game simulation callback).
- Display orientation (vertical/TATE) metadata + rotated presentation in the
  player for the vertical titles.
- No-dump true-M72 protection sets (`dbreedm72`, `dkgensanm72`) declare
  manifest-level `[[hle]]` MCU profiles; current coverage includes the startup
  inverted shared-RAM surface, command-latch acknowledge, and sample-trigger
  cursor setup through profile-bounded trigger tables, with set-specific
  protection entry routines still to validate against authentic artifacts.

## Phase F — M92 (own ADR when M72 is at parity)

V33 as a V30 configuration, GA20 PCM (`chips/audio`), GA21/GA22 video
(`chips/video`), sound-CPU opcode decryption.

## Verification

- Every increment green-builds and green-tests (`ctest`) before commit;
  clang-format clean.
- Existing console suites untouched at every step (no shared code changes
  except the named additive extensions, each landing with its own tests).
- Data-gated suites: V30 singlestep corpus, R-Type boot/attract parity.
